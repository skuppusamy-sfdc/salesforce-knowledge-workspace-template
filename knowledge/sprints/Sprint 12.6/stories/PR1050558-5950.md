## Description

As an SFO, I want the system to perform an additional check for clinician contract mix rates before automatically pushing rates as part of the Client Change Management process, so that we ensure that if any new contracts were added with mixed rate we do NOT auto push rates, but instead create a task for the CET team to review and determine what updates are needed.

This story adds on to story [+PR10505585173+|https://salesforce.atlassian.net/browse/PR10505585173] to cover the new gap scenario that was identified.

## Acceptance Criteria

AC1: Add additional check on Clinician Contracts BEFORE automatically pushing rates

Given I am an SFO in the Corporate Support Apps

When a Practice Operations (PO) Contract is created for Coverage Change = 'Yes'

And Status gets updated to ‘Executed’ 

And system created Client Change Management records with Process Type = ‘Automation’

Then on the PO Contract Start Date, BEFORE auto pushing rates, the system shall:

- Check all the clinician’s contracts in Status = Draft, Submitted, Approved, Contracting - In Progress, or Executed

** Determine if **any* of the Clinician’s Contracts in the above Statuses contain mixed rates:

*** If yes: then create a **task* for CET to manually review and make update all of the clinician contracts as needed [Existing functionality, see stories [+PR10505585173+|https://salesforce.atlassian.net/browse/PR10505585173], [+PR10505585870+|https://salesforce.atlassian.net/browse/PR10505585870] (new, updates 5173 which will add in the contract numbers)]

*** If no: then system **auto pushes rates* into all clinician contracts in Status listed above [Existing functionality, see story [PR10505582916|https://salesforce.atlassian.net/browse/PR10505582916], [+PR10505585173+|https://salesforce.atlassian.net/browse/PR10505585173]]

## Solution

Repeate the Logic to update the Client Change Management Record Process Type = Manual before running the Clinician Change job

The logic is in chained queued job need to move to Apex and the Apex need to be reused in both chained queuable job as well as as the batch job
