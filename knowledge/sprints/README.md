# Sprint Stories Directory

## ⚠️ Important: Supplemental Knowledge Base

**This workspace is SUPPLEMENTAL to your actual code repository.** The code lives in your separate Salesforce repository (managed via Git/Copado). This workspace helps you understand requirements, design solutions, and plan work - but the actual implementation and metadata live elsewhere.

## Organization

Each sprint has its own subdirectory with an HTML export from JIRA and auto-generated per-story markdown files:
```
/knowledge/sprints/
  /Sprint 1/
    SCP Sprint 1.html            # Original JIRA export (monolithic)
    /stories/                    # Auto-generated per-story files
      PR1050558-100.md
      PR1050558-101.md
      ...
  /Sprint 14/
    SCP HC Sprint 14.html
    /stories/
      PR1050558-6425.md
      ...
```

The `stories/` subdirectories are generated by `scripts/split-sprint-stories.py`. Each `.md` file contains the full metadata, description, acceptance criteria, solution, and linked issues for a single story. This enables precise RAG retrieval — the AI fetches one 2-30KB story file instead of parsing a 1-4.5MB sprint HTML.

## HTML File Structure

Your HTML files are JIRA exports containing tables with each row representing a story. Key columns to focus on:

### Critical Fields
- **Issue Key** - JIRA story number (e.g., PR1050558-123)
- **Acceptance Criteria** - What defines "done"
- **Solution** - Technical approach and implementation details
- **Summary** - Story title
- **Description** - Detailed requirements
- **Status** - Current state
- **Components** - Affected Salesforce components
- **Story Points** - Complexity estimate

### How to Use These Files

1. **Ask Cursor to read them**: Cursor can read HTML and extract story information
2. **Focus on AC and Solution**: These fields bridge requirements to implementation
3. **Cross-reference with code**: Use story numbers to find actual code in your repository

## Per-Story Markdown Files

Each sprint now contains a `stories/` subdirectory with one markdown file per JIRA story. These files are the **preferred source** for Cursor AI lookups — they are smaller, faster to retrieve, and contain the full un-truncated content.

### Regenerating Story Files

When you add a new sprint HTML export, generate the per-story files:
```bash
cd scripts/
python3 split-sprint-stories.py --sprint "Sprint N"   # new sprint only
python3 split-sprint-stories.py                        # all sprints
python3 split-sprint-stories.py --force                # overwrite existing
```

## Extracting Information from HTML Files

Cursor AI can read and analyze your HTML files directly. The per-story markdown files are preferred for single-story lookups, while the HTML files remain available for bulk analysis.

## Tips for Best Results

1. **Be specific with story numbers**: Use the full JIRA key (e.g., PR1050558-123)
2. **Reference sprint names as they appear**: "Sprint 14", "Sprint 12.3", etc.
3. **Ask about Acceptance Criteria and Solution explicitly**: These are the most important fields
4. **Request comparisons**: Cursor can compare stories across sprints
5. **Use for planning, not coding**: Remember this workspace is supplemental to your code repository
