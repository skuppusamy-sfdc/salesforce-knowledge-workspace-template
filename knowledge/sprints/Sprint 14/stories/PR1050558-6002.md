## Description

**PERMISSIONS – Support Desk: Experience Cloud Password Reset & User Unlock (High)**

**Current State:**

- Support desk staff cannot reset Experience Cloud passwords or unlock user accounts.

- All such requests escalate to a small pool of system admins, creating significant bottlenecks during hypercare.

**Business Impact:**

- Disproportionate admin burden for routine tasks; SLA risk during hypercare as support volume is high.

**Requirement:**

- Evaluate granting support desk staff the minimum required permissions to reset Experience Cloud passwords and unlock user accounts.

- Determine ownership: should this be Recruiter role, Support Desk role, or a new delegated admin group?

**Suggested Acceptance Criteria:**

-◦     Support desk users can reset Experience Cloud user passwords without admin escalation.-

-◦     Support desk users can unlock Experience Cloud user accounts without admin escalation.-

-◦     Permission changes are scoped to minimum access required — no unnecessary object/field exposure.-

-◦     Change is tested in sandbox with the designated support desk persona before production deploy.-

## Acceptance Criteria

As a Corporate Support user I can Reset Experience Cloud User Passwords so that I can help Clinician Login to the system when they can’t remember their passwords. _– Covered under story_ [+_PR10505585999_+|https://salesforce.atlassian.net/browse/PR10505585999]

As a Corporate Support user I can Unlock Experience Cloud Users Accounts so that if a Clinician has tried the wrong password so many times as to lock their account, I can unlock it.

NOTE: this does NOT include the RUL users

Steps to unlock:

From the Clinician’s record select action = “Unlock User”

A pop up should appear stating “Your request to unlock the user has been submitted.”

Then click “Finish”

## Solution

Expose Only Record Page Account for Clinician

Create a Flow as Button in Person Account Page that will invoke a Platform Event (Run the Platform Even as System User) 

Consume the Platform Event in a Flow that unlocked the User Records

**_Its solution this way to avoid View Setup permission for Users_**
