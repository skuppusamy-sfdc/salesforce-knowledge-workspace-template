# Recipe: Build a RAG-Powered Knowledge Workspace for a Salesforce Agile Project

| Field | Content |
|---|---|
| **Recipe Name** | Build a RAG-Powered Knowledge Workspace for a Salesforce Agile Project |
| **Phase** | Design / Build / Test / Review / Document — used across All phases by SA, TA, Dev, QA |
| **Problem It Solves** | **Today:** Sprint stories live in JIRA. Salesforce metadata (1000+ Apex, Flows, Objects, LWC) lives in repositories. Traceability lives in deployment tools. Conflict detection (Technical & Functional) across sprints is manual and takes hours per sprint. Architects re-read the same large artifacts for every question. Test scripts are created manually, using nothing but the tester's own recollection and experience. The whole delivery depends on "Tribal Knowledge." **With the agent:** A single, read-only context layer fuses stories + metadata + traceability into pre-computed, searchable indexes. Conflict checks drop from hours to seconds. Tribal knowledge becomes documented, instantly queryable knowledge. |
| **Thought Process** | Don't mix your active coding environment with your project knowledge base. Build a **separate, read-only Cursor workspace** dedicated purely to context retrieval. Don't point the LLM at giant raw files (JIRA exports, metadata repos, deployment CSVs) — it's slow and wastes tokens. Instead, pre-process those sources once into lightweight, role-oriented Markdown indexes (Master Story, Component-to-Story Map, AC Index, Traceability Index). Frame the AI's behavior via `.cursor/rules`: *"You are a read-only assistant. Look in the Markdown indexes first. Only open the raw source if the index doesn't have what you need. Always cite your source."* |
| **The Prompt(s)** | **0. Workspace Bootstrap (Given to Agent once):** *"Set up a read-only knowledge workspace. Inputs: JIRA stories in `knowledge/sprints/`, Salesforce QA metadata in `knowledge/metadata/`, traceability in `knowledge/traceability/`. Pre-compute Markdown indexes for: stories, components, AC, solutions, traceability, component-to-story. Add `.cursor/rules/` to enforce index-first lookup and no-code-generation."* |
| | **1. SA — Sprint conflict check:** *"I'm planning a story for `<component>`. Search Component-to-Story Map for prior stories on it, cross-reference AC Index for overlapping criteria, and Traceability Index for recent deployments. Flag technical and functional conflicts. Cite story numbers."* |
| | **2. SA — AC confidence check:** *"For `<JIRA-key>`, list every AC. Find prior stories with similar AC patterns and tell me if the existing implementation satisfies, partially satisfies, or contradicts it."* |
| | **3. TA — Pattern reuse:** *"I need to design `<capability>`. Search Solution Index for prior implementations, group by tech stack, and summarize the pattern in 5 bullets."* |
| | **4. TA — Impact analysis:** *"If I modify `<component>`, what's downstream? List stories that touched it (Component-to-Story Map) and recent deployments (Traceability Index)."* |
| | **5. Dev — 60-second onboarding:** *"Explain `<JIRA-key>`: summary, AC, solution approach, components touched, and Copado mapping. End with a one-paragraph summary."* |
| | **6. Dev — Prior art lookup:** *"I need to build `<feature>`. Search Solution and Master Story Indexes for anything similar already shipped. Show the closest 3 stories."* |
| | **7. QA — Test script generation:** *"For `<JIRA-key>`, pull its AC and related components. Generate a test script outline grouped by AC (positive/negative/regression). Do not invent ACs not in the index."* |
| | **8. QA — Regression scope:** *"For deploying `<JIRA-keys>`, list every component changing and every prior story that touched them. Flag which prior ACs need regression testing."* |
| | **9. Catch-all query:** *"Using the indexes, answer: `<question>`. Cite the index and source file. Don't parse raw sources unless explicitly asked."* |
| **Example Input** | *User (Solution Architect):* "I am planning a new story to add validation rules to the `Provider_Contract__c` object. Search the `COMPONENT-TO-STORY-MAP.md` for prior stories touching this object. Then cross-reference the `AC-INDEX.md` to see if we already have overlapping validation criteria for Provider Contracts. Finally, check the `TRACEABILITY-INDEX.md` to see the last time this object was deployed. Flag any potential conflicts." |
| **Example Output** | *AI Agent:* **Analysis of Provider_Contract__c** — **1. Prior Stories (from COMPONENT-TO-STORY-MAP.md)** Found 3 stories that previously modified `Provider_Contract__c`: PR-4092 (Sprint 12): "Add base contract fields", PR-4155 (Sprint 12.1): "Contract routing automation", PR-4201 (Sprint 12.3): "End-date auto-populate". **2. AC Overlap (from AC-INDEX.md)** Looking at the 'Validation' category for these stories: **Potential Conflict:** PR-4092 established a validation rule: *"User must not be able to save if Effective Date is in the past."* If your new validation rules interact with dates, ensure they do not contradict this existing rule. **3. Deployment Status (from TRACEABILITY-INDEX.md)** `Provider_Contract__c` was most recently deployed in Sprint 12.3 under Copado User Story US-0004512. **Conclusion:** There is a high risk of functional conflict regarding date validations established in Sprint 12. Please review the full AC for PR-4092 before finalizing your new criteria. |
| **Tips & Gotchas** | **For users interacting with this read-only workspace:** |
| | **"Where is my code?"** — It's not here. This is a read-only knowledge repository. Use this workspace to research, plan, and detect conflicts. Write your actual Apex/LWC code in your separate dev IDE. |
| | **Force Index Usage** — The agent can be lazy. If you ask "What's in Sprint 12?" without directing it to an index, it might try to read a massive raw file and time out. Always start prompts with: *"Check the MASTER-STORY-INDEX..."* to guarantee a 5-second answer. |
| | **Beware of Stale Context** — The indexes represent a snapshot of QA/JIRA at the time the Python scripts were last run (usually sprint boundary). If a dev committed code yesterday, it won't be reflected here yet. |
| | **Show Your Work** — Always demand citations. If the agent says "We built a similar flow in Sprint 8," ensure it cites the JIRA key and the index it used. Don't trust uncited claims. |
| | **The Traceability Gap** — Not every JIRA story maps 1:1 to a deployment record. If a story was purely functional (no metadata changed), it won't appear in the Traceability Index. |
| | **"Build Components" Noise** — In JIRA, people type free-text into fields like "Impacted Areas". The index captures this, but it's noisy. Always verify against the actual metadata in the `COMPONENT-CATALOG`. |
| **Works In** | Cursor (primary — Agent + Plan modes). Portable to: Gemini (read-only Q&A over the indexes), NotebookLM (excellent for ingesting the Markdown indexes as a notebook for non-Cursor users), AgentForce / Slackbot. Pre-processing scripts are Python. |
